# Документация кодовой базы Elephants

Elephants - игровой прототип на C++20 и SFML 3. Игрок управляет стадом слонов на гекс-карте: стадо постоянно движется, потребляет растительность, удобряет клетки навозом, растет при достатке еды и постепенно сокращает площадь лесов.

## Быстрый старт

Сборка на Windows рассчитана на CMake и vcpkg manifest mode. Локальный preset `windows-vcpkg` ожидает vcpkg в `C:/vcpkg`.

```powershell
cmake --preset windows-vcpkg
cmake --build --preset windows-debug
.\build\windows-vcpkg\Debug\elephants.exe
```

Тесты запускаются так:

```powershell
cmake --build --preset windows-debug
ctest --test-dir .\build\windows-vcpkg -C Debug --output-on-failure
```

Если vcpkg установлен в другом месте, используйте preset `ninja-vcpkg` с переменной окружения `VCPKG_ROOT` или поправьте `CMakePresets.json`.

## Структура проекта

```text
CMakeLists.txt              CMake targets, warnings, CTest integration
CMakePresets.json           Windows/vcpkg and Ninja/vcpkg configure presets
vcpkg.json                  vcpkg manifest, dependency on sfml
include/Elephants/          public headers for game subsystems
src/                        implementation files
tests/SimulationTests.cpp   lightweight unit test runner
docs/CODEBASE.md            this document
```

CMake собирает три цели:

- `elephants_core`: модельные классы `HexGrid` и `Simulation`. Эта цель не открывает окно и используется тестами.
- `elephants`: приложение с окном SFML, вводом, рендером и игровым циклом.
- `elephants_tests`: простой test executable без внешнего test framework.

## Основной поток выполнения

Точка входа находится в `src/Main.cpp`. Она создает `elephants::Game` и вызывает `Game::run()`.

`Game` отвечает за runtime-обвязку:

- создает `sf::RenderWindow` размером `1280x900`;
- хранит `Simulation` и `Renderer`;
- обрабатывает события SFML;
- передает время кадра в симуляцию;
- очищает окно, вызывает рендер и показывает кадр;
- обновляет заголовок окна статусной строкой из симуляции.

Игровая логика находится в `Simulation`. Рендер не меняет состояние игры, а только читает `Simulation`, `Tile`, `Herd` и `SimulationStats`.

## Модель данных

Основные типы объявлены в `include/Elephants/Types.hpp` и `include/Elephants/Simulation.hpp`.

`TerrainType` описывает тип ландшафта:

- `Forest` - лес;
- `Grassland` - степь;
- `DryGrassland` - сухая степь;
- `Water` - вода, непроходима;
- `Desert` - пустыня;
- `Swamp` - болото.

`HexCoord` хранит axial-координаты `q` и `r`. Для координаты не хранится `s`, но расстояние считается через `s = -q - r`.

`Tile` хранит состояние одного гекса:

- `terrain` - текущий тип ландшафта;
- `rainfall` - уровень осадков от `0` до `1`;
- `fertility` - плодородие;
- `vegetation` - текущее количество растительности;
- `manure` - накопленный навоз.

`Herd` хранит состояние стада:

- `center` - центральный гекс стада;
- `direction` - выбранное направление движения;
- `size` - размер стада;
- `hunger` - сглаженный уровень сытости от `0` до `1`.

`SimulationConfig` содержит балансные параметры. Новые числовые настройки лучше добавлять туда, а не хардкодить в логике.

## Гекс-сетка

`HexGrid` находится в `include/Elephants/HexGrid.hpp` и `src/HexGrid.cpp`.

Карта создается как шестиугольная область заданного радиуса. Количество клеток для радиуса `r`: `1 + 3 * r * (r + 1)`. Например, радиус `2` дает `19` клеток.

Направления в axial-координатах зафиксированы в `HexGrid::directions()`:

```text
{ 1,  0}
{ 1, -1}
{ 0, -1}
{-1,  0}
{-1,  1}
{ 0,  1}
```

`toPixel()` использует pointy-top раскладку гексов:

```text
x = hexSize * sqrt(3) * (q + r / 2)
y = hexSize * 1.5 * r
```

`indexOf()` сейчас ищет координату линейным проходом по `coords_`. Для текущего размера карты этого достаточно. Если карта станет большой, стоит добавить индекс `unordered_map<HexCoord, size_t>` с hash-функцией.

## Симуляция

`Simulation::update(dt)` накапливает время и выполняет один или несколько дискретных тиков, когда накопитель достигает `SimulationConfig::tickSeconds`.

Порядок одного тика:

1. `moveHerd()` - стадо пытается перейти в выбранном направлении.
2. `feedHerd()` - стадо потребляет еду, добавляет навоз и может ломать лес.
3. `regenerateTiles()` - клетки восстанавливают растительность, навоз влияет на плодородие.
4. `updateResult()` - проверка победы или поражения.

### Генерация карты

`generateMap()` вызывает `makeTile()` для каждой координаты. Карта детерминирована относительно координат: шум считается через `smoothNoise(coord, seed)`, без random-device.

Осадки зависят от расстояния до центра, широтной сухости и шума. Затем по rainfall и второму шуму выбирается ландшафт:

- высокий rainfall: лес или болото;
- средний rainfall: лес или степь;
- низкий rainfall: сухая степь, степь или пустыня;
- часть влажных клеток превращается в воду/реку.

Начальная растительность задается как доля от максимальной емкости клетки.

### Движение стада

Игрок задает направление через `Simulation::setDirection()`. Метод принимает только одно из шести направлений `HexGrid::directions()`, неверный вектор игнорируется.

Если следующий гекс в выбранном направлении проходим, стадо переходит туда. Если он непроходим или вне карты, стадо выбирает лучшего проходимого соседа по эвристике:

```text
score = vegetation * foodAccessibility + fertility * 1.5
```

Вода непроходима. Остальные типы ландшафта проходимы.

### Питание, рост и голод

Спрос на еду считается от размера стада:

```text
demand = herd.size * baseConsumptionPerElephant * movementPenalty
```

В лесу применяется штраф движения `1.22`, поэтому стадо расходует больше еды.

Доступная еда на клетке равна:

```text
accessibleFood = vegetation * foodAccessibility(tile)
```

Потребленная еда удаляет часть `vegetation`, увеличивает `manure` и влияет на `lastFoodRatio`. Сытость `herd.hunger` сглаживается, чтобы UI не дергался от одного тика.

Если еды достаточно, стадо растет. Если еды мало, стадо уменьшается. Размер ограничивается диапазоном от `0` до `maxHerdSize`; поражение наступает ниже `minHerdSize` или при почти нулевой сытости.

### Навоз, плодородие и восстановление

В `regenerateTiles()` каждая проходимая клетка:

- получает прирост плодородия от навоза;
- постепенно возвращается к климатическому baseline плодородия;
- теряет часть накопленного навоза;
- восстанавливает растительность к `maxVegetation(tile)` со скоростью `growthRate(tile)`.

Степь имеет лучший темп восстановления и максимальную кормовую ценность. Лес содержит растительность, но хуже доступен как корм. Пустыня почти не кормит.

### Воздействие большого стада

`herdFootprint()` возвращает клетки, на которые действует стадо:

- размер меньше `14`: только центральный гекс;
- размер от `14`: центр и первый радиус соседей;
- размер от `35`: центр, первый радиус и второй радиус по шести направлениям.

Только проходимые клетки попадают в footprint.

### Разрушение леса

При кормлении на лесной клетке есть шанс превратить ее в `Grassland`. Шанс зависит от размера стада:

```text
breakChance = forestBreakBaseChance + herd.size * forestBreakChancePerSize
```

Он ограничен сверху значением `0.35`. После превращения клетка получает небольшой прирост растительности и плодородия.

### Победа и поражение

Победа: доля лесных клеток среди проходимой суши меньше или равна `victoryForestRatio`.

Поражение: размер стада меньше `minHerdSize` или `hunger <= 0.05`.

Пока игра в состоянии `Won` или `Lost`, `Simulation::update()` больше не меняет состояние.

## Рендеринг

`Renderer` находится в `include/Elephants/Renderer.hpp` и `src/Renderer.cpp`.

Рендер полностью процедурный, внешние ассеты не требуются:

- гексы рисуются через `sf::ConvexShape`;
- базовый цвет зависит от `TerrainType`;
- внутренний overlay показывает относительную растительность;
- footprint стада подсвечивается контуром;
- стадо рисуется кругами: тень, тело и хобот, ориентированный по направлению движения;
- HUD содержит три шкалы: сытость, прогресс уменьшения леса, средняя растительность.

`Renderer` хранит ссылку на `HexGrid`. Поэтому `Simulation` не должен заменять объект `grid_` после создания `Renderer`. Текущий `Simulation::reset()` пересоздает только тайлы и стадо, а сетка остается той же.

## Управление

Клавиши задаются в `Game::handleKeyPressed()`:

```text
E -> { 1,  0}
W -> { 1, -1}
Q -> { 0, -1}
A -> {-1,  0}
S -> {-1,  1}
D -> { 0,  1}
Space -> пауза
R -> новая карта
Esc -> выход
```

`R` вызывает `Simulation::reset()` и снимает паузу.

## Тесты

Тесты находятся в `tests/SimulationTests.cpp`. Это lightweight runner без GoogleTest/Catch2, чтобы не добавлять лишние зависимости.

Покрытые сценарии:

- размер и соседи `HexGrid`;
- валидность начальной генерации карты;
- проверка допустимых и недопустимых направлений;
- один тик симуляции меняет экологическое состояние и сохраняет инварианты;
- footprint увеличивается с размером стада;
- высокий порог победы переводит игру в `Won` после update.

Если добавляется новая модельная логика, предпочтительно покрывать ее здесь, потому что `elephants_tests` не открывает окно и быстро запускается через CTest.

## Как расширять код

### Новый тип ландшафта

Чтобы добавить ландшафт, нужно обновить:

- `TerrainType`;
- `Simulation::makeTile()`;
- `Simulation::maxVegetation()`;
- `Simulation::growthRate()`;
- `Simulation::foodAccessibility()`;
- `Renderer::tileColor()`;
- тесты на генерацию/инварианты при необходимости.

### Новый параметр баланса

Добавляйте поле в `SimulationConfig`, затем используйте его в `Simulation`. Это сохраняет возможность быстро создавать тестовые конфигурации без изменения глобальных констант.

### Новый UI-индикатор

Данные для UI лучше отдавать через `SimulationStats` или отдельный read-only accessor в `Simulation`. Не стоит читать и агрегировать весь `tiles()` в `Renderer`, если показатель относится к игровой логике.

### Ассеты вместо процедурной графики

Текущий `Renderer` не зависит от файлов. Если появятся PNG-спрайты, стоит добавить fallback: сначала пытаться загрузить ассет, а при ошибке использовать текущую процедурную отрисовку. Это сохранит запуск тестовых и dev-сборок без asset pipeline.

## Инварианты и соглашения

- `Simulation` владеет игровым состоянием; `Renderer` состояние не меняет.
- Вода непроходима; все остальные текущие ландшафты проходимы.
- Значения `hunger`, `forestRatio`, `averageVegetation`, `herdFoodRatio` нормализованы в диапазон `0..1`.
- `tiles_` и `grid_.coords()` должны иметь одинаковый размер и общий индекс.
- `Simulation::update()` не делает ничего после победы или поражения.
- Публичные accessors возвращают const-ссылки или значения; прямого внешнего изменения модели сейчас нет.

## Известные ограничения

- Генерация карты детерминирована и не меняется между `reset()`, потому что noise seed фиксирован.
- `HexGrid::indexOf()` линейный, что нормально для текущего радиуса, но не для больших карт.
- HUD пока без текста внутри окна: подробный статус выводится в заголовок окна.
- Визуальный спрайт стада процедурный, не ассетный.
- Тесты покрывают модельные инварианты, но не проверяют визуальный рендер SFML.
